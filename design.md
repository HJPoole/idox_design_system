# Idox Design System

## Overview

The Idox Design System is the shared visual and interaction language for Idox Software's product suite — applications serving local authorities, planning departments, licensing teams, and public sector professionals across the UK. The system is built for clarity, reliability, and accessibility: the people using these interfaces are working professionals who need to complete tasks accurately, often under time pressure, and often within heavily regulated workflows.

The aesthetic is functional and confident — clear hierarchy, restrained use of colour, and deliberate structure. The system does not compete for attention; it supports the work. Visual decisions are governed by usability and accessibility requirements first, and aesthetic considerations second.

The target audience is public sector professionals, council officers, and case workers. The emotional response sought is competence, trust, and ease — users should feel that the interface is in control and working for them.

## Colors

The palette is organised into two tiers: primitive colour values (the raw scale) and semantic tokens (purposeful assignments). Components always reference semantic tokens, never primitive values directly.

### Surfaces

Three surface levels create depth in the layout without asserting colour.

- **surface-default (#FFFFFF):** The primary component surface — panels, cards, input backgrounds, modals. Pure white for maximum contrast against content.
- **surface-base (#F8FAFC — slate.50):** The page canvas and secondary fill — slightly off-white backgrounds, panel header fills, sidebar backgrounds. Gives components visual lift relative to the page.
- **surface-muted (#E2E8F0 — slate.200):** Used for skeleton loader placeholders and heavily muted fills where the absence of content needs to be communicated visually.

### Text

- **text-primary (#0F172A — slate.900):** All primary body copy, headings, and labels.
- **text-secondary (#475569 — slate.600):** Supporting text — subheadings, metadata, form field labels. Achieves clear visual hierarchy below the primary level.
- **text-tertiary (#94A3B8 — slate.400):** Hint text, timestamps, and placeholder descriptors. Lowest text contrast in the hierarchy — use only for truly supplementary information.
- **text-inverse (#FFFFFF):** Text on dark surfaces — navbar, tooltips.
- **text-brand (#0A1F8F — blue.800):** Brand-coloured text — headings in brand contexts.
- **text-brand-accent (#195FD2 — blue.600):** Accent text and the focus outline colour for interactive elements.

### Borders

Four border levels cover every structural need.

- **border-subtle (#F1F5F9 — slate.100):** The lightest separator — internal panel dividers, table row dividers, section breaks within components.
- **border-default (#E2E8F0 — slate.200):** The standard outer component border — panels, cards, tables. Light and recessive.
- **border-strong (#94A3B8 — slate.400):** Used where structural clarity is critical — input fields, select menus, strong section dividers.
- **border-focus (#9CC2FC — blue.300):** The focus ring colour applied via `ring-focus` on inputs and form controls when keyboard-focused.
- **border-selected (#195FD2 — blue.600):** The border colour applied to selected items — selected list items, active tabs, and chosen option indicators.
- **border-accent (#195FD2 — blue.600):** Accent border for brand-themed elements — primary buttons and elements that need to assert brand presence through their border.

### Links

- **link-default (#195FD2 — blue.600):** Standard inline link colour.
- **link-hover (#1241B2 — blue.700):** Link hover state.
- **link-active (#0A1F8F — blue.800):** Link active/pressed state.

### Interactive

The interactive scale drives all primary action affordances — buttons, toggle states, selected rows, active navigation items.

- **interactive-default (#0A1F8F — blue.800):** Default fill for primary interactive elements — primary buttons, active links, checkboxes, radio buttons.
- **interactive-hover (#1241B2 — blue.700):** Hover state — one step lighter from the default, maintaining contrast on white surfaces.
- **interactive-active (#050F48 — blue.900):** Pressed/active state — communicating tactile feedback.
- **interactive-text (#FFFFFF):** Text and icons placed on any interactive fill colour.
- **interactive-subtle (#F5F9FE — blue.50):** Tinted background for toggle-active states and persistent active indicators.
- **interactive-subtle-hover (#ECF3FE — blue.100):** Hover background for ghost elements and icon-only buttons.
- **interactive-subtle-active (#D2E4FE — blue.200):** Pressed background for subtle interactive elements and active sidebar navigation items.

### List States

- **list-hover-bg (#F8FAFC — slate.50):** Background applied on hover to list items, dropdown options, and table rows.
- **list-active-bg (#ECF3FE — blue.100):** Background applied on press to list items and dropdown options.

### Buttons

Button tokens are distinct from the general interactive scale to allow independent theming of button fills versus other interactive elements.

- **btn-primary / btn-primary-hover / btn-primary-active:** Navy interactive scale — fills for the primary button at rest, hover, and pressed states.
- **btn-primary-text (#FFFFFF):** Label text on primary buttons.
- **btn-secondary (#FFFFFF):** White fill for secondary buttons.
- **btn-secondary-hover / btn-secondary-active:** Subtle blue tints applied on hover and press.
- **btn-secondary-text (#0F172A):** Label text on secondary buttons — primary text colour for maximum contrast on white fill.

### Navbar

- **navbar-bg (#0A1F8F):** Navbar background — the same deep navy as the brand interactive colour.
- **navbar-text (#FFFFFF):** Primary navbar text and active nav item labels.
- **navbar-hover / navbar-active:** Hover and active fills for navbar links and icon buttons.
- **navbar-icon (#D2E4FE — blue.200):** Default icon colour in the navbar — light blue-tinted against the dark navy background.
- **navbar-icon-hover (#FFFFFF):** Icon colour on hover.

### Status

Status colours follow a four-value pattern across all three states: `default` (filled indicator), `subtle` (alert/callout background), `border` (alert outline), and `content` (inline text colour). All status colour combinations meet WCAG AA on their respective backgrounds.

- **Success:** Green family (green.600 default / green.50 subtle / green.300 border / green.700 content)
- **Error:** Red family (red.600 default / red.50 subtle / red.300 border / red.700 content)
- **Warning:** Amber family (amber.500 default / amber.50 subtle / amber.300 border / amber.800 content)

Warning content text uses amber.800 rather than white because amber.500 fill is too light to achieve sufficient contrast with white text.

## Typography

The system uses a single typeface: **DM Sans**, a low-contrast humanist sans-serif optimised for screen legibility. DM Sans renders clearly at small sizes — critical in data-dense interfaces where labels, table cells, and helper text must be readable without strain.

In practice, the UI type scale is compact. Most interactive and form UI uses `text-sm` (14px) as the primary reading size, with `text-xs` (12px) for fine print and `text-base` (16px) reserved for panel section headings. The larger heading values (h1–h3) are available for page-level headings and marketing contexts.

### Line Heights

Line heights follow the semantic scale: `tight` (1.25) for headings, `normal` (1.5) for body text and labels, `relaxed` (1.625) for longer prose. Do not apply `tight` line heights to body-sized text — it reduces legibility in paragraph form.

## Layout

The layout system is built on a centred container model with a `1280px` maximum width and `24px` gutters on smaller viewports. This cap suits the wide monitor setups common in local authority office environments while maintaining readable line lengths.

The spacing scale follows a 4px base unit across all components and layout elements. Internal component padding uses `spacing.3` (12px) to `spacing.6` (24px). Section-level vertical rhythm uses `spacing.12` (48px) to `spacing.24` (96px).

Application layouts follow a fixed-height shell: top navbar, optional breadcrumb bar, then a flex row of sidebar and scrollable content area. The sidebar is `w-56` (224px) and can be toggled. The content area fills the remainder and scrolls independently.

Data-heavy views (tables, forms, case records) should prioritise horizontal scanability. Use sufficient row height to prevent touch-target failures and reduce dense-layout fatigue.

## Elevation & Depth

Depth is expressed through shadow and border in combination. The system provides a structured shadow scale:

- **shadow-sm:** Subtle lift for input fields and inline components — distinguishes them from the page surface without asserting visual weight.
- **shadow-default / shadow-md:** Card and panel elevation — used for floating UI elements like dropdowns and date pickers.
- **shadow-lg / shadow-xl:** Modal overlays and drawers — maximum elevation within the viewport.

Cards use `shadow-sm` at rest. Hover states elevate to `shadow-md` and strengthen the border from `border-default` to `border-strong`. This two-signal approach (shadow + border) ensures interaction feedback is never dependent on a single visual property — important for users who may adjust display contrast.

Modal overlays use `slate.950` at 50% opacity as the scrim behind the modal container.

## Shapes

The border radius scale runs from `none` (0px) to `full` (9999px). Interactive components — inputs, buttons, icon buttons — use `rounded.lg` (8px) as the default. Panels and cards use `rounded.xl` (12px). Pill badges and full-circle indicators use `rounded.full`.

The scale is intentional: the moderate rounding (lg/xl) gives the system a modern, approachable character while remaining professional and legible. Extreme rounding (`3xl`, `full`) is reserved for badges, avatars, and UI elements that need to be immediately distinguishable from structural containers.

## Focus Management

Two focus ring patterns are used depending on context:

- **Inputs, selects, and textareas:** `focus:ring-1 focus:ring-focus focus:border-focus` — a 1px ring using `border-focus` (#9CC2FC) drawn over the existing border.
- **Buttons, links, and toolbar controls:** `focus-visible:outline-solid focus-visible:outline-2 focus-visible:outline-offset-2 focus-visible:outline-brand-accent` — a 2px solid outline drawn outside the element boundary using `text-brand-accent` (#195FD2).

Both patterns satisfy WCAG 2.2 SC 2.4.11 Focus Appearance. The split exists because outline-offset creates cleaner focus presentation on elements with fills (buttons), while the ring approach is better integrated into the input border treatment.

## Components

### Navigation

The top **Navbar** uses `navbar-bg` fill with `navbar-text` for links and labels. Icon-only action buttons use `navbar-icon` at rest and `navbar-icon-hover` on hover. The navbar height is `h-14` (56px). It contains a logo, product name, primary navigation links, and utility icon buttons.

The **Side Navigation** panel is `w-56` (224px) and uses `surface-base` as its background. Navigation links use `nav-link` tokens at rest and `nav-link-active` for the current route. Active items display `interactive-default` text and `interactive-subtle-active` background. Nested navigation levels are indented. Section labels use `overline` typography in `text-tertiary`. The sidebar can be toggled via a toolbar button.

A **Breadcrumb** bar sits between the navbar and the main content area — `h-9`, `surface-default` background, `border-subtle` bottom border. It displays the current path and optional utility controls (fullscreen toggle, information button).

### Buttons

Four button variants cover all interactive contexts:

- **Primary:** `btn-primary` fill, `btn-primary-text` label, `border-brand-accent` border. The single most-important action in a view. Used once per major context.
- **Secondary:** `btn-secondary` fill, `btn-secondary-text` label, `border-strong` border. Supporting actions alongside a primary. The border is `border-strong` — not `border-brand-accent` — to keep secondary buttons visually recessive relative to primary.
- **Ghost:** `bg-transparent`, `text-primary` label, no border. Low-emphasis actions within dense layouts where a bordered button would add too much visual noise.
- **Disabled:** `opacity-50`, `cursor-not-allowed` on a secondary-styled button. Never use `button-destructive` as a distinct variant — destructive actions are always secondary-styled and always paired with a confirmation dialog.

All buttons use `rounded-lg` (8px) and `text-sm font-semibold`. Minimum height is 32px (`h-8`) for compact toolbar contexts; standard buttons use `py-1.5` padding. Leading icons use `text-base` size on the `<i>` element. Icon-only buttons are `w-8 h-8`.

### Tabs

Tabs sit at panel level and use `border-b-2 border-brand-accent` on the active tab with `text-primary font-semibold`. Inactive tabs use `text-secondary` with a hover state that applies `text-primary`. Tabs do not have background fills — they rely solely on the bottom border indicator.

### Form Fields

Input fields follow a four-state model: default, focused, error, and disabled. Default uses `border-strong`. Focused adds `ring-1 ring-focus` and `border-focus`. Error replaces the border with `border-2 border-red-600`. Disabled applies `opacity-50 cursor-not-allowed`.

Field labels use `text-sm font-medium text-secondary`. Helper text uses `text-sm text-tertiary`. Error messages use `text-sm font-medium text-red-600` and are always accompanied by `fa-circle-exclamation`.

Every input must have a visible label — placeholder text alone is not sufficient. Labels are positioned above the input.

Prefix inputs use an absolutely-positioned icon or text character within the input wrapper, with left padding on the input to accommodate it. Inputs with action buttons (date pickers, clear controls) use an absolutely-positioned button flush to the right edge of the input wrapper.

### Toolbar

The **Toolbar** is a horizontal strip of controls — `surface-default` background, `border-default` border, `rounded-lg`, `px-3 py-2`. It groups icon-only buttons with `w-px h-5 bg-default` vertical separators between logical groups. Labelled buttons (e.g. Save) can appear at the left edge. Icon-only formatting toggle buttons use `aria-pressed` for active state and apply `bg-interactive-subtle text-interactive-default` when active.

### Panel

**Panels** are the primary surface for grouping related content. Structure: `section.bg-surface-default.border.border-default.rounded-xl.overflow-hidden`. The header is `px-5 py-3 border-b border-subtle` with a `text-base font-semibold text-primary` heading. The body is `p-5`. Use `overflow-hidden` on the outer container to clip the header border radius correctly. Panels can be made collapsible using a `<details>` element.

### Data Tables

Tables are wrapped in a panel. The search bar and pagination controls sit in the panel header at `px-5 py-3 border-b border-subtle`. The table itself sits in `overflow-x-auto`. Column headers use `bg-surface-base border-b border-subtle px-5 py-2 text-sm font-semibold text-secondary`. Body rows use `divide-y divide-subtle` and `hover:bg-surface-base`. Row height minimum is 40px.

Sortable columns display `fa-light fa-caret-down` at rest. Active ascending sort uses `fa-solid fa-caret-up` with `aria-sort="ascending"`. Active descending sort uses `fa-solid fa-caret-down` with `aria-sort="descending"`.

Custom checkboxes use a visually-hidden native `<input type="checkbox">` with peer utilities for checked and focus states. The visible indicator is a styled `<span>` that responds to `peer-checked:` and `peer-focus-visible:` states.

Row action menus use an icon-only ellipsis button (`fa-regular fa-ellipsis`) with `w-7 h-7`.

### Callout Panels

Inline callouts surface contextual information, warnings, or errors within a page. Three semantic variants:

- **Info:** `bg-blue-50 border border-blue-200`, icon `fa-circle-info text-blue-500`.
- **Warning:** `bg-amber-50 border border-amber-200`, icon `fa-triangle-exclamation text-amber-500`, heading `text-amber-900`, body `text-amber-800`.
- **Error:** `bg-red-50 border border-red-200`, icon `fa-circle-exclamation text-red-500`, heading `text-red-900`, body `text-red-800`.

All callouts use `flex gap-3 rounded-lg px-5 py-3`. When used inside a two-column form grid, they should span `md:col-span-2`.

### Dialog

Dialogs use a fixed overlay (`bg-slate-950/50 z-40`) and a centred modal container (`max-w-xl rounded-lg border border-default bg-surface-default shadow-lg`). Structure: header (`px-6 pt-6` with title and close button), body (`px-6 py-6`), footer (`px-6 pb-6` with action buttons left-aligned). The close button is a `w-7 h-7` icon-only button using `fa-light fa-xmark`.

Focus must be trapped within the dialog while it is open. `Escape` closes the dialog. Focus returns to the triggering element on close. The trigger button carries `aria-haspopup="dialog"`, `aria-controls`, and `aria-expanded`.

### Dropdown Menu

Dropdown menus use the `<el-dropdown>` / `<el-menu>` web components from Tailwind Elements. The trigger is a secondary button with a `fa-solid fa-chevron-down` indicator. The menu panel is `w-60 rounded-lg border border-default bg-surface-default p-1 shadow-lg mt-2`. Items are full-width buttons with `flex items-center gap-2 px-3 py-2 rounded-md text-sm`. Destructive items use `text-red-700`. Separators use `<hr class="my-1 border-subtle">`.

### Dual Listbox

The dual listbox presents two `<select multiple>` elements side by side with transfer buttons between them. Options use `[&>option]:px-3 [&>option]:py-1` padding and `[&>option:checked]:bg-interactive-default [&>option:checked]:text-interactive-text` for selection state. Transfer buttons are secondary buttons with chevron icons.

### List Builder

The list builder uses a tree structure (`role="tree"`, `role="group"`) in two panes with transfer buttons between them. Expandable sections use `<details>/<summary>`. Selectable leaf items are `<button>` elements with class `tree-item`. The selected item carries class `is-selected` with `background: var(--interactive-default)` locked via CSS to prevent hover/active override.

### Autocomplete / Lookup

The autocomplete uses an input with an absolutely-positioned dropdown list below it. The list is `absolute z-30 w-full rounded-lg border border-default bg-surface-default shadow-lg p-1`. Each result is a full-width button using list item hover/active states. A clear button sits flush to the right edge of the input wrapper.

### Tooltip

Tooltips use the `data-tooltip` attribute and CSS `::after` pseudo-element. The tooltip appears to the right of the element. Background is `idox-slate-800` (#1e293b), text is white, `font-size: 0.875rem`, `font-weight: 500`, `border-radius: 0.3rem`, `padding: 0.3rem 0.5rem`. Delay is handled via CSS `transition: opacity 0.15s`.

### Checkboxes and Radio Buttons

Both use visually-hidden native inputs with `peer` utility classes controlling visible indicator states. The visible wrapper uses `peer-checked:bg-interactive-default peer-checked:border-interactive-default` for the filled state and `peer-focus-visible:ring-2 peer-focus-visible:ring-focus` for keyboard focus. Checkboxes show `fa-regular fa-check text-xs text-inverse` when checked. Radio buttons show a `h-1.5 w-1.5 rounded-full bg-white` inner dot when selected. Disabled state uses `opacity-50 cursor-not-allowed`.

### Date Picker

The date picker consists of an input with a calendar toggle button, and a panel that opens below. The panel (`w-72 rounded-xl border border-default bg-surface-default shadow-lg`) contains a month navigation header, a 7-column day grid, and footer links (Today, Clear). Today's date uses `ring-1 ring-interactive-default text-interactive-default font-semibold`. The selected date uses `bg-interactive-default text-interactive-text` with `aria-pressed="true"`. Weekend columns use `text-secondary` instead of `text-primary`.

### Spinner

Spinners use `animate-spin` on an SVG circle with a partial fill — `opacity-25` for the track circle and `opacity-75` for the progress arc. Always include `role="status"` and a visually-hidden "Loading…" label. Include `motion-reduce:animate-none` to respect the user's reduced-motion preference. Sizes: default `w-8 h-8`, small `w-5 h-5`, inline-in-button `w-4 h-4`. Colours: default uses `text-interactive-default`; muted uses `text-tertiary`.

### Skeleton Loader

Skeleton loaders use `animate-pulse motion-reduce:animate-none` on `bg-surface-muted` placeholder shapes. The wrapper carries `role="status"` and `aria-label="Loading content"` with a visually-hidden "Loading…" span. Shapes mirror the real content layout — rectangular blocks for text lines, button-height blocks for action areas. Prefer skeletons over spinners when the final layout is known in advance.

### Empty State

Empty states are used when a list, table, or panel has no content. Centred layout: a large icon at `text-4xl text-tertiary`, a short heading at `text-sm font-semibold text-primary`, a supporting sentence at `text-sm text-secondary`, and an optional primary action button. The icon uses `aria-hidden="true"`.

## Do's and Don'ts

- **Do** use DM Sans exclusively. No second typeface is introduced, including for code — use the monospace stack only for `code` and `pre` elements, or for structured reference numbers within content.
- **Do** reference semantic tokens in components — never primitive colour values (e.g. `#0A1F8F`) or Tailwind primitives (e.g. `blue-800`) directly.
- **Do** ensure all interactive elements have a visible focus state — `focus:ring-focus` on inputs, `focus-visible:outline-brand-accent` on buttons and controls.
- **Do** use `text-secondary` and `text-tertiary` only for genuinely subordinate information — if something is important, use `text-primary`.
- **Do** apply `opacity-50` to disabled components — never change the layout or collapse the space of disabled elements.
- **Do** use status colours only for their intended semantic meaning. Error red must not be used for branding; warning amber must not be used decoratively.
- **Do** maintain minimum touch targets of 32×32px on all interactive elements (icon-only buttons are `w-8 h-8`).
- **Do** validate all status colour combinations against WCAG AA (4.5:1 minimum for text) before shipping.
- **Don't** use `text-tertiary` for interactive affordances — it does not meet WCAG AA contrast requirements and signals non-interaction.
- **Don't** use the brand blue for decorative purposes — fills, gradients, or pattern elements. Blue communicates interactivity.
- **Don't** use `border-brand-accent` on secondary buttons — secondary buttons use `border-strong`. Only primary buttons carry the brand-accent border.
- **Don't** create new status colours outside the success / error / warning / neutral taxonomy. If a new state is needed, map it to the nearest existing semantic group and record the mapping.
- **Don't** truncate form labels or helper text to save space. If a field label is too long for a layout, the layout is too narrow for that content — adjust the layout.
- **Don't** use `border-radius` values outside the defined scale. The scale covers every use case in the system.
- **Don't** use arbitrary Tailwind value syntax (`bg-[#4F46E5]`, `mt-[13px]`) — raise as a token gap instead.
